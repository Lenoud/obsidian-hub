# 集群角色绑定

集群角色绑定相关技术笔记。

```yaml
#kubectl create clusterrolebinding --clusterrole cluster-admin --serviceaccount default:sa-nfs clusterrolebinding-nfs --dry-run -oyaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  creationTimestamp: null
  name: clusterrolebinding-nfs
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: cluster-admin
subjects:
- kind: ServiceAccount
  name: sa-nfs
  namespace: default

```

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: gitlab-cluster-admin
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: cluster-admin
subjects:
- apiGroup: rbac.authorization.k8s.io
  kind: Group
  name: system:serviceaccounts
```

这段配置文件描述了创建了一个名为gitlab-cluster-admin的集群角色绑定(ClusterRoleBinding)，用于将Kubernetes集群管理员角色授权给所有命名空间中的ServiceAccount。

具体来说，它的作用是：

+ 通过roleRef字段指定了该集群角色绑定的角色(ClusterRole)，这里是默认的Kubernetes集群管理员角色cluster-admin。
+ 通过subjects字段指定了需要授权的身份。在这里，指定了system:serviceaccounts组内的所有ServiceAccount，即所有命名空间中的ServiceAccount都将被授权为集群管理员。
+ 由于使用的是ClusterRoleBinding，因此该集群角色绑定对整个Kubernetes集群中的所有命名空间都有效。

总之，这段配置文件的作用是将集群管理员角色授权给所有命名空间中的ServiceAccount，使它们具有访问整个集群资源的最高权限。这通常用于需要在整个集群范围内管理资源的特殊任务或场景。
