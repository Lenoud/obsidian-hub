# 集群角色绑定

集群角色绑定相关技术笔记。

使用 ClusterRoleBinding 将 ServiceAccount 或 Group 绑定到 cluster-admin 集群管理员角色，赋予集群范围内的最高权限。

## 绑定 ServiceAccount

```yaml
#kubectl create clusterrolebinding --clusterrole cluster-admin --serviceaccount default:sa-nfs clusterrolebinding-nfs --dry-run -oyaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  creationTimestamp: null
  name: clusterrolebinding-nfs
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: cluster-admin
subjects:
- kind: ServiceAccount
  name: sa-nfs
  namespace: default
```

## 绑定所有 ServiceAccount

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: gitlab-cluster-admin
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: cluster-admin
subjects:
- apiGroup: rbac.authorization.k8s.io
  kind: Group
  name: system:serviceaccounts
```

这段配置文件描述了创建了一个名为 `gitlab-cluster-admin` 的集群角色绑定（ClusterRoleBinding），用于将 Kubernetes 集群管理员角色授权给所有命名空间中的 ServiceAccount。

具体来说，它的作用是：

- 通过 `roleRef` 字段指定了该集群角色绑定的角色（ClusterRole），这里是默认的 Kubernetes 集群管理员角色 `cluster-admin`。
- 通过 `subjects` 字段指定了需要授权的身份。在这里，指定了 `system:serviceaccounts` 组内的所有 ServiceAccount，即所有命名空间中的 ServiceAccount 都将被授权为集群管理员。
- 由于使用的是 ClusterRoleBinding，因此该集群角色绑定对整个 Kubernetes 集群中的所有命名空间都有效。
