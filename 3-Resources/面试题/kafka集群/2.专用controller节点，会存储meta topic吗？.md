在 Kafka 的 **KRaft 模式**下，**专用 Controller 节点（仅配置 **`process.roles=controller`**）会存储元数据（即 **`__cluster_metadata`** 主题）**，但不会存储用户数据（普通 Topic 的分区数据）。以下是关键细节：

---

### **1. 专用 Controller 节点的职责**
+ **仅处理元数据**：
专用 Controller 节点不承担普通 Broker 的角色，因此：
    - **不存储用户数据**：普通 Topic 的分区数据（如生产者的消息）不会存储在这些节点上。
    - **专注于元数据管理**：通过 Raft 协议参与元数据（如 Topic 配置、分区分配、ACL 规则等）的共识和同步。
+ **存储元数据日志和快照**：
    - 元数据以日志形式存储在 `__cluster_metadata` 主题中，由所有 Controller 节点共同维护。
    - 每个 Controller 节点在本地磁盘的 `log.dirs` 目录下的 `meta/` 子目录中存储：
        * **元数据日志**：记录所有元数据变更操作（如创建 Topic、分区重分配）。
        * **元数据快照**：定期生成的压缩快照，用于快速恢复。

---

### **2. 数据存储位置对比**
| **节点类型** | **存储内容** |
| --- | --- |
| **专用 Controller 节点** | 仅存储元数据（`meta/` 目录下的 `__cluster_metadata` 日志和快照）。 |
| **Broker 节点** | 存储用户数据（普通 Topic 的分区数据，位于 `log.dirs` 目录的 Topic 子目录）。 |
| **混合角色节点** | 同时存储元数据（`meta/`）和用户数据（`log.dirs`）。 |

---

### **3. 为什么专用 Controller 节点需要存储元数据？**
+ **Raft 协议的要求**：
KRaft 模式依赖 Raft 共识算法管理元数据，**所有 Controller 节点必须存储完整的元数据日志和快照**，以达成多数派（Quorum）共识。若 Controller 节点不存储元数据，将无法参与 Leader 选举或日志复制，导致集群元数据不可用。
+ **高可用性保障**：
    - 元数据的多副本存储（在多个 Controller 节点上）防止单点故障。
    - 即使部分 Controller 节点宕机，只要剩余节点满足多数派（例如 3 个节点中存活 2 个），集群仍能正常处理元数据变更。

---

### **4. 配置示例**
若部署一个由 3 个专用 Controller 节点组成的 Quorum，配置如下：

```properties
# 专用 Controller 节点配置（不存储用户数据）
process.roles=controller
node.id=1 # 每个节点唯一
controller.quorum.voters=1@controller1:9093,2@controller2:9093,3@controller3:9093
log.dirs=/var/lib/kafka/metadata # 仅存储元数据
```

---

### **5. 注意事项**
1. **磁盘隔离**：
建议为元数据（`meta/`）配置独立的磁盘，避免与用户数据（如有混合角色节点）的 I/O 竞争。
2. **最少节点数**：
生产环境至少部署 **3 个 Controller 节点**，以容忍 1 个节点故障。
3. **角色分离**：
专用 Controller 节点无需高性能磁盘（元数据操作远少于用户数据读写），但需保证低网络延迟，以快速达成 Raft 共识。

---

### **总结**
在 KRaft 模式下，**专用 Controller 节点会存储元数据（**`__cluster_metadata`** 主题）**，这是 Raft 协议实现强一致性和高可用性的基础。而用户数据仅由 Broker 节点（或混合角色节点）存储。这种分离设计提升了 Kafka 的可扩展性和运维灵活性。
