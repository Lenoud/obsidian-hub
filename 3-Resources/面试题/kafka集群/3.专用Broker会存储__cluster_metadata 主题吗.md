在 Kafka 的 **KRaft 模式**下，**专用 Broker 节点（仅配置 **`process.roles=broker`**）不会存储 **`__cluster_metadata`** 主题**。以下是详细说明：

---

### **1. 专用 Broker 节点的职责**
+ **仅处理用户数据**：
专用 Broker 节点不参与元数据管理，因此：
    - **不存储元数据**：`__cluster_metadata` 主题的日志和快照不会存储在这些节点上。
    - **专注于用户数据存储和读写**：负责存储普通 Topic 的分区数据（生产者和消费者处理的消息），并处理客户端的读写请求。
+ **元数据访问**：
专用 Broker 节点会从 Controller 节点（通过 Raft 协议管理元数据）获取所需的元数据信息（如 Topic 配置、分区分配等），但不会存储这些元数据。

---

### **2. 数据存储位置对比**
| **节点类型** | **存储内容** |
| --- | --- |
| **专用 Broker 节点** | 仅存储用户数据（普通 Topic 的分区数据，位于 `log.dirs` 目录的 Topic 子目录）。 |
| **专用 Controller 节点** | 仅存储元数据（`meta/` 目录下的 `__cluster_metadata` 日志和快照）。 |
| **混合角色节点** | 同时存储元数据（`meta/`）和用户数据（`log.dirs`）。 |

---

### **3. 为什么专用 Broker 节点不存储 **`__cluster_metadata`** 主题？**
+ **职责分离**：
KRaft 模式通过角色分离（`process.roles`）明确节点职责：
    - **Controller 节点**：负责元数据管理，存储 `__cluster_metadata` 主题。
    - **Broker 节点**：负责用户数据存储和读写，不参与元数据管理。
+ **性能优化**：
    - 元数据操作（如创建 Topic、分区重分配）通常远少于用户数据读写操作。
    - 将元数据管理集中在少量 Controller 节点上，可以降低 Broker 节点的负载，提升集群整体性能。
+ **简化运维**：
    - 专用 Broker 节点无需维护元数据日志和快照，减少了磁盘和网络开销。
    - 元数据的高可用性由 Controller 节点通过 Raft 协议保证，无需 Broker 节点参与。

---

### **4. 配置示例**
若部署一个专用 Broker 节点，配置如下：

```properties
# 专用 Broker 节点配置（不存储元数据）
process.roles=broker
node.id=4 # 每个节点唯一
controller.quorum.voters=1@controller1:9093,2@controller2:9093,3@controller3:9093
log.dirs=/var/lib/kafka/data # 仅存储用户数据
```

---

### **5. 注意事项**
1. **Controller 节点的高可用性**：
专用 Broker 节点依赖 Controller 节点获取元数据，因此需确保 Controller 节点的高可用性（至少 3 个节点）。
2. **网络延迟**：
Broker 节点与 Controller 节点之间的网络延迟应尽可能低，以避免元数据访问成为性能瓶颈。
3. **角色分离的优势**：
专用 Broker 节点可以专注于用户数据存储和读写，适合大规模数据处理的场景。

---

### **总结**
在 KRaft 模式下，**专用 Broker 节点不会存储 **`__cluster_metadata`** 主题**，这是由 Kafka 的职责分离设计决定的。元数据由专用 Controller 节点通过 Raft 协议管理，而 Broker 节点专注于用户数据的存储和读写。这种设计提升了 Kafka 的性能、可扩展性和运维灵活性。
